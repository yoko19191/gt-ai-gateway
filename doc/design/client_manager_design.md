# 客户端管理配置逻辑讨论稿

本文档记录「客户端管理」模块的目标配置逻辑，用于后续实现和讨论。第一版仅支持 Claude Code 和 Codex。

## 核心原则

客户端管理页面中的配置列表必须满足：

1. **每一行配置都对应数据库中的一条配置记录。**
2. 前端不展示“游离于数据库之外”的配置行。
3. 勾选状态严格来自数据库字段 `client_config.enabled`。
4. 本地客户端配置文件只用于检测当前实际状态，不直接作为列表中的一行。
5. 创建配置只创建数据库记录，不写入本地客户端配置文件，不自动启用。

也就是说，页面上的“第一行配置”也应该是数据库配置，而不是单独的“当前本地配置”虚拟行。

## 数据模型

表：`client_config`

| 字段 | 说明 |
|------|------|
| `id` | 配置记录 ID |
| `client` | 客户端类型，例如 `claude-code`、`codex` |
| `name` | 配置名称 |
| `configContent` | 提取后的配置参数（如 gatewayUrl、apiKey、model 等），JSON 字典结构 |
| `enabled` | 是否启用 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

约束：

- 同一个 `client` 下最多只能有一条 `enabled = true` 的配置。
- 每个前端配置卡片都来自这张表。
- `enabled` 是前端勾选状态的唯一来源。

## 本地配置文件的角色

本地配置文件不是配置列表中的独立行。

它只用于：

1. 检测客户端是否已安装。
2. 检测本地文件是否已被配置。
3. 和当前启用配置做内容对比，判断是否“被修改”。

本地文件读取来源：

- Claude Code：`~/.claude/settings.json`
- Codex：`$CODEX_HOME/config.toml`、`$CODEX_HOME/auth.json`

## 页面结构

每个客户端 tab 内只展示数据库配置记录列表。

如果当前客户端没有任何数据库配置记录：

- 配置列表显示为空状态。
- 不展示“当前本地配置”虚拟行。
- 可以显示“新配置”和“从本地配置新建”按钮，引导用户创建第一条配置。

每个配置卡片展示：

- 配置名称
- 是否启用
- 模型
- 连接模式
- 详情按钮
- 修改名称入口
- 删除入口
- 切换/启用入口

如果存在启用配置，并且本地配置文件与启用配置不一致：

- 启用配置卡片显示“被修改”。
- 页面可以在客户端标题区域显示整体状态提示。

不再单独展示“当前本地配置”这一行。

## 勾选状态逻辑

勾选状态必须严格对应后端字段：

- `backup.enabled = true`：前端显示已勾选。
- `backup.enabled = false`：前端显示未勾选。

前端不能通过“本地文件内容是否等于某条配置”自行推导勾选状态。

后端返回：

- `activeBackupId`：当前启用配置 ID，从数据库 `enabled` 字段推导。
- `activeConfigModified`：当前本地配置是否相对启用配置发生变化。

## 切换配置

用户点击某条配置前面的未勾选圆圈时：

1. 后端读取该配置记录的 `configContent`（包含 url, key, model 等抽象参数）。
2. 后端读取当前磁盘上实际的客户端配置文件（以便保留原本手动配置的内容，例如 mcpServers）。
3. 后端用配置记录中的 `configContent` 参数去 Patch (覆盖) 磁盘文件中的相关字段。
4. 将合并后的新内容写回对应客户端的本地配置文件。
5. 将同一 `client` 下所有配置记录的 `enabled` 设为 `false`。
6. 将当前配置记录的 `enabled` 设为 `true`。
7. 返回更新后的客户端配置状态。

切换完成后：

- 当前配置记录显示勾选。
- 其它配置显示未勾选。
- 当前本地配置应与启用配置一致，因此 `activeConfigModified = false`。

## 被修改状态

如果某个客户端存在启用配置，后端会比较：

1. 从当前本地配置文件中解析提取出的配置参数 (Parsed Fields)。
2. 启用配置记录中保存的 `configContent`。

比较结果：

- 内容一致：`activeConfigModified = false`。
- 内容不一致：`activeConfigModified = true`。

前端展示：

- 在启用配置卡片上显示“被修改”。
- 可在客户端标题区域显示提示，说明本地配置文件已和当前启用配置不一致。

这个状态表示：数据库中标记启用的配置，已经不再完全等于磁盘上的真实配置文件。

## 创建配置

创建配置必须产生数据库记录。

### 新配置

用户点击“新配置”：

1. 弹窗选择连接方式、用户/供应商、模型等。
2. 后端生成对应客户端配置内容。
3. 从生成的配置文件中提取出参数，创建一条 `client_config` 记录，保存到 `configContent`。
4. 新记录的 `enabled = false`。
5. 不写入本地客户端配置文件。
6. 不影响当前客户端实际行为。

因此，“新配置”完成后页面列表中会出现一条新的未启用配置。只有用户显式点击该配置的勾选/启用入口时，才会写入本地客户端配置文件并设置为启用。

### 从本地配置新建

页面提供“从本地配置新建”按钮。

用户点击后：

1. 后端读取当前本地配置文件。
2. 从当前本地文件内容中提取出配置参数，使用这些参数创建一条新的 `client_config` 记录。
3. 新记录的 `enabled = false`。
4. 不写入本地客户端配置文件。
5. 不影响当前客户端实际行为。

该功能用于把用户已有的本地客户端配置导入到数据库配置列表中，后续可以重命名、修改或显式启用。

## 修改配置

配置修改也应该作用于数据库记录。

推荐流程：

1. 用户点击某条配置的“修改”。
2. 弹窗基于该配置记录的内容初始化表单。
3. 用户提交后，后端更新该配置记录的 `configContent`。
4. 如果该配置是启用配置，则后端读取当前磁盘文件，使用新的配置参数进行 Patch，并将合并后的结果同步写回本地客户端配置文件，从而保留文件中其他与网关无关的自定义配置（如 mcpServers）。
5. 如果该配置不是启用配置，则只更新数据库记录，不影响本地文件。

这样可以保持：

- 每个卡片都对应数据库配置。
- 启用配置和本地文件保持一致。
- 非启用配置可以预先编辑，但不会影响当前客户端行为。

## 删除配置

每条数据库配置记录都支持删除。

用户点击配置卡片的删除入口时：

1. 前端弹出确认框，提示删除不可恢复。
2. 后端删除对应的 `client_config` 记录。
3. 返回更新后的客户端配置状态。

删除非启用配置：

- 只删除数据库记录。
- 不影响本地客户端配置文件。
- 不影响当前启用配置。

删除启用配置：

- 删除数据库记录。
- 不自动修改本地客户端配置文件。
- 同一 `client` 下不再有启用配置。
- 后端返回 `activeBackupId = undefined`。
- 后端返回 `activeConfigModified = false`，因为已经不存在可对比的启用配置。

删除启用配置后，本地客户端仍然可能继续使用原来的文件内容，但该文件内容已经不再对应任何启用配置。

## 连接模式展示

配置解析后会显示连接模式：

1. **代理模式**
   - 客户端连接 GT AI Gateway。
   - Gateway 再连接后端供应商。
   - 如果 token 能匹配到网关用户，详情中显示对应用户。

2. **直连模式**
   - 客户端直接连接上游供应商。
   - 不显示网关用户。
